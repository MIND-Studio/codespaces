**Real root cause found — staying `ready-for-human` until it's verified in prod.**

Two layers were in play. The in-process **session cache** (`src/lib/solid/oidc-server.ts`,
`tests/oidc-session-cache.test.ts`) shipped as v0.1.19 and stops re-deriving/burning the
fetch — but it doesn't *create* a refresh token that was never minted. The actual prod
cause is upstream: the bridge's scripted CSS consent POST never asked CSS to **remember**
the grant, and CSS only issues a refresh token for a remembered consent. Fix (on `main`,
not yet committed):

```
src/lib/solid/css-account.ts
- const consent = await reqJson(consentUrl, jar, { method: "POST", body: {} });
+ const consent = await reqJson(consentUrl, jar, { method: "POST", body: { remember: true } });
```

So `startAuthFlow` *does* request `offline_access` (AC2) — the missing piece was the
remembered consent. **Owed before close (AC1/AC3/AC4):** commit the `remember: true` fix,
deploy the bridge to `codespaces.mindpods.org`, re-`/connect` hermesbot once (its stored
identity has no refresh token), then confirm a fresh-connect WebID's `publishPages()`
renders `…/public/sites/<repo>/` 200 while an older identity still publishes. Not done yet.
